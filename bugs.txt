THOUGHTS:

- the waiting issue after some commands, can it be related to signals? since it gets interrupted by pressing enter, what process does enter interrupt or what signal is blocked/ignored? 

- alternative builtin-logic could be to check for builtin commands after tokenizing and compare the tokens

- main: after calling builtinsmenu, use continue to restart the loop without going further, instead of if/else run_cmd

- ft_execvp is an implementation of execvp. Instead of changing the logic in the function, maybe better to make a separate function that calls it. Now the function does a lot of different things, which makes it hard to trouble shoot. 

- run_cmd is part of the main loop, it gets called by handle_child_process, handle_parent_process, handle_redir_cmd. It's part of the recursive logic. It never returns and only exits on error or when a command is finished executing. Forking is done in main in every iteration of the loop (for each command). Creating more forks in run_cmd might cause undefined behaviour and zombie kids.

- it's adviced against using signal() in general, since it's an unstable function and specially when running parallell processes, such as piping. If problems with waiting etc persists, we might wanna consider switching to sigaction.
https://stackoverflow.com/questions/231912/what-is-the-difference-between-sigaction-and-signal
